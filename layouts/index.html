Documentation GitHub

go-embedded-ruby compiles Ruby to bytecode and runs it on a stack VM in the mruby/YARV lineage, with the lexer, parser and compiler embedded in the binary so eval and runtime require keep working. Ruby objects are Go heap objects, so Go's garbage collector is reused — there is no GC to write. Because it is pure Go with cgo disabled it cross-compiles trivially, and it is the first Ruby you can embed inside a Go program with no C toolchain.

Pure Go with cgo disabled, so it cross-compiles to a single static binary. A bytecode VM with the front-end embedded keeps Ruby's dynamism — eval, runtime require, monkey-patching, method_missing — at your fingertips. Ruby objects are Go heap objects, so the Go garbage collector is reused. The standard library is selected at build time and tree-shaken. Semantics track Ruby 4.0, grown test-first against an MRI 4.0.5, JRuby (and TruffleRuby) differential oracle, with 100% coverage enforced in CI across all six 64-bit architectures (amd64, arm64, riscv64, loong64, ppc64le, s390x) and three OSes. Two core pieces are reusable pure-Go modules in sibling orgs, imported by the interpreter (and usable by any Go program without cgo): the front-end — lexer, parser and AST — is go-ruby-parser, and the regexp engine is a pure-Go reimplementation of Onigmo in go-ruby-regexp.

Beyond the language, a cgo-free scientific stack ships as built-in Ruby classes, each binding a pure-Go sibling library: NDArray (NumPy-style n-dimensional arrays), an FFT module (numpy.fft-style transforms, no FFTW), Image (scikit-image-style processing), and the container/value types Set, Bag, Time, Date and BigDecimal (from go-composites) — alongside Complex, Rational and Math.

WebAssembly is a first-class target. Because everything is pure Go with cgo disabled, the interpreter and the whole stack compile to GOOS=js GOARCH=wasm and run entirely in the browser — there is no server-side code. It ships two ways: a self-contained playground links the full interpreter (front-end included) so a page can evaluate arbitrary Ruby — a REPL plus an image pipeline (load → blur/edges → render) — while rbgo build --closed --target wasm app.rb cross-compiles a closed-world wasm app that runs one program with no front-end linked and drives the page's DOM and Canvas through the built-in JS module (JS.document, JS.log, JS::Ref#call, JS.raf) — a Ruby browser app with no hand-written JavaScript.

Conformance. Correctness is judged against independent reference implementations of Ruby 4.0 — MRI (CRuby) 4.0.5, JRuby and TruffleRuby — with a four-way differential oracle (scripts/oracle.sh runs a snippet through all three and flags any divergence). Beyond synthetic tests, the bar is real-world Ruby. Confronted with the two largest reference codebases, the pure-Go front-end (parse + compile) now accepts 99.96% of Rails + Puppet (5577/5579 .rb files — the only 2 misses are intentional syntax-error fixtures MRI itself rejects, i.e. 100% of all valid Ruby); end-to-end parse + compile is Rails 99.82% and Puppet 100%, with 0 over-permissive. Popular libraries parse just as well — RuboCop 99.7%; Sinatra, Jekyll, Thor, Kramdown and dry-struct 100% — and RSpec DSL usage is 10/10 byte-identical to MRI. These are front-end "parses and compiles" figures, not whole-application execution: running a full app additionally needs the runtime stdlib surface and C-extension equivalents. That surface is now real enough to boot a real application — see Running Puppet below — though running any given app end-to-end remains application-specific work.

Running Puppet. Beyond parsing real-world Ruby, rbgo now runs the real Puppet puppet apply CLI end-to-end: require "puppet" fully boots the framework (Puppet 8.11.0) on a pure-Go CGO=0 rbgo — its pure-Ruby gem dependencies (semantic_puppet, concurrent-ruby, facter, …) load on the $LOAD_PATH — and a manifest then travels the complete Puppet path: the genuine Puppet::Util::CommandLine → Puppet::Application::Apply entry point (real OptionParser), every type + provider loaded, the settings catalog applied to disk, then the user catalog applied through the transaction / RAL. The real CLI emits genuine Puppet output and exits 0 — apply -e 'notify { "hello": message => "hi from rbgo cli" }' prints Notice: hi from rbgo cli and Notice: …/Notify[hello]/message: defined 'message' as 'hi from rbgo cli'. Reaching the CLI added a real OptionParser, File::Stat/FileTest + on-disk filesystem ops, and deep Ruby fixes (class_eval lexical scope, return in define_method, class-method super, …) — validating the C-extension → pure-Go shim strategy that ships real Ruby apps as one static CGO=0 binary. Honest frontier: what runs end-to-end is the notify resource applying through the transaction; real file/package/service resource providers (host convergence) and run-report persistence are the active next milestone, not done.

Performance. A 6-runtime comparative suite pits rbgo and rbgo+AOT against MRI, MRI+YJIT, JRuby and TruffleRuby on the same programs (output checked byte-identical first). The interpreter runs ~3–6× MRI on compute-bound code and at parity where startup / I-O dominates; rbgo+AOT beats MRI+YJIT 18–24× on loop/fib — the only runtime here that beats YJIT — by lowering integer-bound methods to native Go, while TruffleRuby (GraalVM JIT) is the compute ceiling.
